SLIDE 4: TECHNICAL ARCHITECTURE (3:30 - 5:30)
═══════════════════════════════════════════════════════════════

🏗️ HOW DID WE BUILD IT?

Architecture:
  ├─ Game State (snake, food, score)
  ├─ Game Loop (update, render, handle input)
  ├─ Collision Detection (walls, self, food)
  └─ Graphics Rendering (console characters)

Core Data Structures:
  • Snake: Array of positions + direction
  • Food: Position + active flag
  • Game: Centralized state management

Key Algorithms:
  • Movement: Ring buffer pattern
  • Collision: O(n) array check
  • Difficulty: Frame-based speed control

SPEAKER NOTES (120 seconds):
"Our architecture is modular and clean. We separated concerns into
 distinct functions:

 The game manages state - where the snake is, where food is, what's
 the current score. The game loop runs every frame: first we handle
 input from the player, then we update the game logic, then we
 render the display.

 For the snake, we use an array to store position history. When
 the snake moves, we add a new head and remove the tail. This gives
 us a smooth movement without dynamic allocation.

 Collision detection checks:
  1. Did we hit a wall? (Boundary check)
  2. Did we hit ourselves? (Array comparison up to body length)
  3. Did we hit food? (Single position check)

 The rendering uses ASCII characters - @ for head, o for body,
 * for food. It's simple but surprisingly effective!

 Difficulty scaling: we count frames and move every X frames.
 As the player scores, we decrease X, making the snake faster."

SPEAKER NOTES (90 seconds):
"We took code quality very seriously. We compiled with strict flags:
 -Wall to catch all warnings, -Wextra for extra checks, -std=c99 for
 standards compliance.

 The result? Zero warnings, zero errors. The code compiles clean.

 For memory management, we made a deliberate choice: no dynamic
 allocation. Every data structure - the snake, the game state,
 everything - is stack-allocated with fixed sizes. This means:
  • No malloc = no memory leaks
  • No pointers = fewer bugs
  • No fragmentation = better performance

 The maximum snake length is 100 segments. That's 800 bytes of memory.
 The entire game state is under 2KB. Compare that to typical games
 that use megabytes just for initialization!

 Lines-wise, we wrote very efficient code. 313 lines of pure logic
 (excluding comments). Every line matters."

Team Insights:
✓ C is powerful for performance-critical applications
✓ Good design prevents bugs better than debugging
✓ Test continuously, not just at the end
✓ Clean code is worth the extra effort

SPEAKER NOTES (60 seconds):

 First, we learned the importance of careful design. We didn't just
 start coding - we planned our data structures, our function
 decomposition, our algorithm approach. This preparation paid off.

 Second, we discovered that low-level control matters. By managing
 our own memory and avoiding dynamic allocation, we created a
 program that's faster, safer, and easier to debug than code that
 relies on automatic memory management.

 Third, we understood why testing matters. By testing collision
 detection, movement, and scoring as we went, we caught bugs early.

 Most importantly, we proved that you can build a real, playable,
 professional application in C with just 313 lines of clean code."

PRESENTATION TIPS FOR DELIVERY
═══════════════════════════════════════════════════════════════

How to Present:

1. PRACTICE FIRST
   Read through this entire script once before presenting
   Time yourself - aim for 10-15 minutes
   Practice transitions between slides

2. SPEAK CLEARLY
   Make eye contact with audience
   Don't read directly from script - use it as notes
   Vary your tone (don't be monotone)
   Speak at a natural pace

3. SHOW ENTHUSIASM
   You built something cool - convey that!
   Smile when talking about the project
   React authentically during the game demo

4. DIVIDE SPEAKING DUTIES
   Designate one person per 1-2 slides (share the load)
   Marouan: Overview + Architecture
   Hamza: Features + Demo
   Driss: Learning + Conclusion

5. HANDLE THE LIVE DEMO
   Test the game BEFORE presenting
   Know exactly what you'll show
   Have a backup recording if live demo fails
   Keep it to 1-2 minutes

6. ANTICIPATE QUESTIONS
   Read the Q&A section above
   Prepare answers in your own words
   Be honest if you don't know ("Great question, let me research that")

7. WRAP UP STRONGLY
   End with GitHub link so people can see the code
   Thank the audience
   Invite questions
   Smile!
